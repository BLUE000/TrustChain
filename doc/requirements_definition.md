# TrustChain 要件定義書 (Requirements Specification)

本ドキュメントは、C++アプリケーション向けのビルド時改ざん検知、オンライン認証、およびコピーライト出力（ウォーターマーク）を統合管理するセキュリティフレームワーク「**TrustChain**」の要件定義書です。

---

## 1. システム概要と目的

本フレームワーク「TrustChain」は、開発者が作成したC++アプリケーションが「改ざんされずに公式ビルドとして配布されているか」を判別・追跡し、非公式なビルドや改ざんに対して適切なウォーターマーク表示や動作制限を行うことを目的とします。

---

## 2. システム全体フロー

```
[ 開発・ビルド時 ]
  CMakeLists.txt 実行
       │
       ├─① GitHub リモートマスタのコミットハッシュを取得
       │  (通信エラー、権限不足、不一致の場合は「改ざん/非公式」と判定)
       │
       ├─② TransCipher API (Webサーバー) に「トークン発行依頼」を送信
       │  (is_official フラグ、システム名、ビルド秘密鍵、ハッシュを送信)
       │
       ▼
  コンパイル & リンク
       │ (判定フラグ BUILD_IS_CUSTOMIZED と APIトークン をマクロとして注入)
       ▼
  ビルド成果物 (可搬性のある実行ファイル)


[ アプリケーション起動・実行時 ]
  アプリ起動
       │
       ├─③ TransCipher API に「トークンチェック依頼」を送信 (オンライン認証)
       │
       ├─[検証結果：トークン無効 (不正・期限切れなど)]
       │   └─► 🚨 即座にエラーを表示してアプリを強制終了する
       │
       ├─[検証結果：オフライン or 通信エラー or 改ざん判定(BUILD_IS_CUSTOMIZED=1)]
       │   └─► ⚠️ ウォーターマーク表示ルート (コピーライト強制表示、画面制限など)
       │
       └─[検証結果：正規かつオンライン認証成功]
           └─► ✅ 通常起動 (公式ビルドとしてフル機能を提供)
```

---

## 3. 機能要件 (Functional Requirements)

### F-1: ビルド時の改ざん検知およびトークン発行

#### F-1.1: GitHub リモートリポジトリとの照合による改ざん検知
* **処理タイミング**: CMakeによる構成（Configure）フェーズ。
* **判定ロジック**: 
  1. インターネット接続を確認し、GitHubの対象リポジトリに対して**ローカルHEADのコミットハッシュが存在するか**を問い合わせる。
  2. ローカル環境の最新コミットハッシュおよび作業ディレクトリの状態（未コミットの変更の有無）を取得する。
  3. **非公式判定**: 以下のいずれかに該当する場合、非公式ビルド（`BUILD_IS_CUSTOMIZED = 1`）と判定する。
     * 未コミットのローカル変更がある（`git status --porcelain` が空でない）。
     * ローカルの最新コミットハッシュが、リモートリポジトリ（GitHub）上に存在しない。
     * ネットワークがオフラインである、または権限不足により GitHub から情報を取得できない。

#### F-1.2: 認証サーバーへの「トークン発行依頼」
* **処理内容**: CMakeのビルドスクリプトから `TransCipher` Web API（トークン発行エンドポイント）へ POST リクエストを送信し、APIトークンを取得する。
* **送信データ（JSON）**:
  * アプリケーション識別名 (`system_name`)
  * ビルド秘密鍵 (`build_secret`)
  * ビルドハッシュ (`build_hash`)
  * 公式ビルドフラグ (`is_official`: 上記 F-1.1 の判定結果に基づき `true`/`false`)
* **受信データ**: サーバーから返却されたトークンを抽出し、コンパイルマクロ `TRANSCIPHER_API_TOKEN` としてコンパイル時にバイナリへ埋め込む。

---

### F-2: 起動時のオンライン認証

#### F-2.1: トークンチェック依頼
* **処理タイミング**: アプリケーションのエントリーポイント（起動直後）。
* **処理内容**: 埋め込まれた `TRANSCIPHER_API_TOKEN` を用いて、`TransCipher` Web API（有効無効判定エンドポイント）へ同期（またはブロッキング方式の非同期）通信を行い、トークンの状態を問い合わせる。
* **オフライン時の処理**: 通信タイムアウト、またはオフライン環境の場合は「認証失敗（ウォーターマーク表示ルート）」とする。

#### F-2.2: 認証結果に基づく挙動の制御
サーバーからのレスポンスおよびビルドマクロに基づき、アプリの動作を以下の3パターンに制御する。

| 判定条件 | 分岐ルート | アプリケーションの挙動 |
| :--- | :--- | :--- |
| **トークンが無効（不正・期限切れなど）** | **強制終了ルート** | 致命的なエラーダイアログ（「このビルドは無効化されました」等）を表示し、速やかにプロセスを安全に強制終了（`exit`）する。 |
| **オフライン / 通信エラー / 改ざん判定 (`BUILD_IS_CUSTOMIZED = 1`)** | **ウォーターマーク表示ルート** | アプリケーションは起動するが、画面に消去不可能なコピーライト表記（ウォーターマーク）を強制出力する。 |
| **トークンが有効 ＆ 公式ビルド (`BUILD_IS_CUSTOMIZED = 0`)** | **通常起動ルート** | 制約なしで通常通り起動する。 |

---

### F-3: ウォーターマーク（コピーライト）出力

#### F-3.1: コピーライト強制表示
* **対象**: 「ウォーターマーク表示ルート」に遷移したビルド。
* **表示内容**: 指定された著作権表示（例: `© BLUE000 (Original Creator)`）および非公式ビルドである旨（例: `Custom Build`）を、ユーザーが隠蔽・消去できない形で画面に強制表示する。

---

### F-4: バイナリ透かし（Overlay Watermarking）

#### F-4.1: 独立したツールによる物理的証明の埋め込み
* **処理タイミング**: ビルド完了後。
* **処理内容**: コンパイル済みの実行ファイルに対し、専用の外部ツール（`BinMarkManager`）を用いて、ファイルの末尾（オーバーレイ領域）に暗号化された出自証明および平文のコピーライトを物理的に埋め込む。ビルドパイプライン（CMake）への自動組み込みは行わず、手動またはセキュアなCI環境でのみ実行する。
* **目的**: ソースコードが改変された場合でも、バイナリを解析・復号することで「真の作者」を証明する手段を提供する。

## 4. 非機能要件 (Non-Functional Requirements)

### N-1: セキュリティ・隠蔽性
* **秘密情報の隠蔽**: Discord Webhook URL などのサーバー側で管理される機密情報は、C++クライアントのソースコードやバイナリ内に一切記述・露出させない。すべて認証サーバー（さくらサーバーのTransCipher）側で処理を完結させる。

### N-2: 信頼性・操作性
* **高速な検証処理**: 起動時のオンライン認証は、ユーザーに体感的なストレスを与えないよう、高速なネットワークレスポンス（タイムアウト設定は数秒以内）で行う。
* **オフラインフォールバック**: 悪意のあるユーザーがホストファイルを書き換える等で認証通信を物理的に遮断した場合でも、自動的に「ウォーターマーク表示ルート」に倒すことで、無制限なクリーン使用を防止する。

### N-3: 移植性と保守性
* **単一のインターフェース**: 本機能群をフレームワーク化し、別のC++プロジェクトに対しても最小限のステップでセキュリティ機能をインポート・統合できるように設計する。
